# Proposal: Add H2O PDF Artifact Package

## Intent

Add a PDF-first H2O Allegiant artifact package for wastewater BD field agents. The deliverable is not a Markdown artifact surface; the deliverable is a set of branded, authenticated PDF files generated by AI SDK tools from typed payloads and rendered with a TypeScript PDF renderer.

The current PR1/PR2 foundation remains useful for owner/thread scoping, typed payload persistence, request-scoped tools, and authenticated routes. This proposal pivots the product contract from Markdown-first to PDF-first:

- Field Brief PDF
- Playbook PDF
- Analytical Read PDF
- Proposal Shell PDF

Markdown may remain as an optional/internal mirror later, but it is not the primary V1 artifact output.

## Why Pivot

The user clarified the product intent: the agent should generate real PDF files, inspired by `H2O Allegiant Discovery Agent v3/h2o-allegiant-brand-brand.py` and the older SecondStream Python agent that generated PDF attachments through tools.

The PDF-first direction is supported by:

- `H2O Allegiant Discovery Agent v3/README.md` — v3 product shape is a four-artifact opportunity package.
- `H2O Allegiant Discovery Agent v3/ARCHITECTURE_NOTES.md` — v2 on-demand/Markdown mirror friction was replaced by v3 always-PDF package behavior.
- `H2O Allegiant Discovery Agent v3/h2o-allegiant-brand-brand.py` — canonical ReportLab brand kit and PDF layout primitives.
- `H2O Allegiant Discovery Agent v3/Reference/render_field_brief.py` — thin renderer consuming reusable brand primitives.
- `/Users/ricardoaltamirano/Developer/SecondStream/backend/app/agents/chat_agent.py` — older working precedent where typed PDF tools render, upload, persist, and return attachment metadata.
- `/Users/ricardoaltamirano/Developer/SecondStream/backend/app/services/pdf_renderer.py` — older renderer boundary: typed payload → template renderer → PDF bytes.
- `docs/agent-audit-and-artifact-plan.md` §6 — planned stack is `@react-pdf/renderer`, server-side in Next routes, with `renderToStream`.

## Goals

- Generate real PDF files for all four H2O artifacts.
- Use typed artifact payloads as renderer input; do not ask the model to emit Markdown as the artifact.
- Render PDFs in TypeScript/Next using `@react-pdf/renderer` unless implementation spike proves a blocker.
- Port visual primitives conceptually from `h2o-allegiant-brand-brand.py`, not Python runtime code.
- Store or expose durable PDF output so the user receives downloadable files, not just text.
- Keep artifacts owner-scoped and thread/kind-scoped.
- Preserve Lambda-safe filesystem Agent Skills.
- Keep implementation reviewable by slicing renderer primitives and artifact types.

## Non-Goals

- Do not use Python/ReportLab in production runtime.
- Do not make Markdown the primary artifact deliverable.
- Do not add a preview panel before PDF downloads work.
- Do not add section-by-section streaming as a requirement for V1.
- Do not inject the full Knowledge Base in V1.
- Do not produce fake PDF URLs or claim PDF support before a renderer exists.

## Revised Scope

### Already Useful Foundation

The PR1/PR2 work can be retained as foundation where compatible:

- `loadSkill` Lambda-safe filesystem Agent Skills.
- `Artifact` owner/thread/kind payload persistence.
- request-scoped artifact tool context.
- strict Zod payload schemas.
- authenticated download route skeleton.
- Lambda Artifact table env/IAM and AppSync compatibility fields.

### Corrective PDF-First Scope

1. Add `@react-pdf/renderer` and a renderer boundary.
2. Add H2O PDF brand primitives in TypeScript:
   - brand colors and stage/severity colors,
   - typography/page constants,
   - logo/text fallback,
   - page footer,
   - cover blocks,
   - stage badge,
   - insight box,
   - section headers,
   - cost-of-alternative table,
   - kill-risk card,
   - action card,
   - playbook theme/question components,
   - analytical evidence/table components,
   - proposal scope/commitment components.
3. Implement four PDF documents:
   - `FieldBriefDocument`
   - `PlaybookDocument`
   - `AnalyticalReadDocument`
   - `ProposalShellDocument`
4. Update tools so successful artifact results include PDF format and download URL.
5. Update the PDF route to return `application/pdf` instead of PR1 `501`.
6. Treat Markdown as optional/debug; remove Markdown-only assumptions from tests and tool descriptions.
7. Validate visual structure against the H2O v3 Field Brief reference before considering Field Brief done.

## Acceptance Criteria

- [ ] Each artifact tool returns a real PDF download URL for its artifact kind.
- [ ] PDF downloads return `application/pdf` with attachment disposition and owner checks.
- [ ] Tool outputs do not advertise Markdown as the primary artifact format.
- [ ] Payloads remain structured JSON and are consumed directly by PDF renderers.
- [ ] The PDF renderers use TypeScript/Next and do not depend on Python/ReportLab runtime.
- [ ] Field Brief PDF includes the v3-required structure: cover, stage badge, flag/header support, four sections, insight boxes, cost-of-alternative table, kill-risk cards, and action cards.
- [ ] Playbook, Analytical Read, and Proposal Shell produce branded PDFs with their own structures.
- [ ] No preview panel, section streaming, or full KB injection is required for V1 acceptance.

## Risks

| Risk                                                | Likelihood | Mitigation                                                                                                                                        |
| --------------------------------------------------- | ---------: | ------------------------------------------------------------------------------------------------------------------------------------------------- |
| PDF renderer scope balloons                         |       High | Port only necessary v3 primitives; keep document renderers thin.                                                                                  |
| `@react-pdf/renderer` bundling incompatibility      |     Medium | Run an early spike/test in Next route context before broad port.                                                                                  |
| Proposal Shell customer-facing quality risk         |     Medium | Produce a basic branded PDF but keep visual sign-off as a gate before external use.                                                               |
| Existing Markdown-first tests lock wrong behavior   |       High | Update tests to assert PDF-first tool results and PDF route behavior.                                                                             |
| Lambda cold start grows if rendering in chat Lambda |     Medium | Prefer rendering PDFs in Next download route or a dedicated server route, not chat Lambda, unless generation-time storage is chosen deliberately. |

## Success Criteria

The user can ask the H2O agent for artifacts and receive four real downloadable PDFs, styled according to the H2O brand reference and scoped to the authenticated thread owner.
