# Design: H2O PDF Artifact Package

## Scope and Pivot

This design supersedes the earlier Markdown-first plan. The user clarified that H2O artifacts should be real PDF files generated by tools. Markdown is not the product deliverable.

The desired V1 artifact package contains four PDFs:

1. Field Brief
2. Playbook
3. Analytical Read
4. Proposal Shell

The implementation keeps useful PR1/PR2 foundation — typed payloads, owner/thread scoping, request-scoped tools, authenticated routes — but pivots output behavior to PDF-first.

## Architecture Decisions

| Decision                                             | Alternatives considered                                                          | Rationale                                                                                                                                                                                        |
| ---------------------------------------------------- | -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Use `@react-pdf/renderer` for V1 PDF rendering       | Python/ReportLab, WeasyPrint, browser/Chrome HTML-to-PDF, pdf-lib manual drawing | `docs/agent-audit-and-artifact-plan.md` §6 already selected `@react-pdf/renderer`: Node-native, no Chrome dependency, `renderToStream`, Lambda/Next friendly, built-in Helvetica.                |
| Render PDFs from typed payloads                      | Ask model for Markdown/HTML and convert                                          | Reliable branded PDFs require structured components, not post-hoc parsing.                                                                                                                       |
| Keep Python as visual reference only                 | Run ReportLab in production                                                      | Current app is TS/Next/AWS; Python runtime adds packaging/ops complexity.                                                                                                                        |
| PDF-first tool results                               | Markdown-first with later PDF                                                    | User intent and v3 references require PDFs as the artifact files.                                                                                                                                |
| Store PDF bytes or render on demand with PDF route   | Only JSON + Markdown route                                                       | The PDF route must return `application/pdf`. The final storage strategy may be on-demand rendering or generated-and-stored PDF bytes, but tool output must advertise only real PDF availability. |
| Field Brief visual gate first, then complete package | Ship all four with no visual validation                                          | Field Brief has strongest reference and should validate brand primitives used by follow-ons.                                                                                                     |

Use `@react-pdf/renderer` unless a small spike proves incompatibility.

## PDF Delivery Strategy

The minimum acceptable V1 route behavior:

```text
GET /api/threads/[threadId]/artifacts/[kind]/pdf
  -> authenticate owner
  -> verify thread ownership
  -> load active ready Artifact payload
  -> render PDF from typed payload
  -> return application/pdf with Content-Disposition attachment
```

This keeps JSON as the source of truth and avoids blob consistency issues. If render time becomes a problem, a later lazy cache can store rendered PDFs without changing the payload/tool contract.

If generation-time storage is chosen instead, then tools must write PDF bytes to private storage and persist `pdfStoragePath`, `pdfSizeBytes`, and `pdfMediaType`. That is valid, but it reintroduces blob consistency concerns and should only be chosen if on-demand `@react-pdf/renderer` is too slow or incompatible.

## Tool Contract

Each artifact tool should return a PDF-first result:

```ts
type ArtifactToolResult = {
  artifactId: string;
  artifactType:
    | "field-brief"
    | "playbook"
    | "analytical-read"
    | "proposal-shell";
  title: string;
  status: "ready";
  formats: Array<{
    format: "pdf";
    mediaType: "application/pdf";
    filename: string;
    downloadUrl: string;
  }>;
};
```

Markdown can be added as an optional second format only if the user explicitly wants extraction mirrors. It should not be the only or primary format.

## Trigger Policy

The v3 product direction favors a four-PDF opportunity package. Therefore:

- Attachment with opportunity context triggers generation guidance for all four artifact tools, Field Brief first.
- Package trigger phrases should also request all four PDFs.
- Focused conversational turns still produce text only.

If four separate tool calls become brittle or expensive, consider a single orchestrating tool `generateOpportunityPackage` that accepts one structured package and produces all four artifacts. The user previously requested one tool per artifact, so keep four tools unless implementation shows orchestration is safer.

## Verification Plan

- Renderer smoke tests: each document renderer returns PDF bytes/stream with `%PDF` or valid PDF response headers.
- Route tests: `pdf` returns `application/pdf`; wrong owner and missing artifact remain denied.
- Tool tests: each tool result includes PDF only and no fake Markdown-primary result.
- Visual gate: Field Brief Prairie fixture is structurally/visually compared against the v3 reference intent.
- Build checks: `bun run test`, `bunx tsc --noEmit`, `bun run check`, `bun run verify:amplify-config`, and sandbox after backend/package changes.
